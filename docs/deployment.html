Deployment

Running the service locally and with Docker, plus the full environment-variable reference. Behavior is configured entirely by environment variables; there is no config file.

Prerequisites

  • Python 3.10 (pinned via .python-version; managed by uv automatically).
  • uv for dependency management.
  • FFmpeg, Sox, and libsndfile on PATH.
  • CUDA 13.0 for GPU acceleration (PyTorch wheels come from the cu130 index configured in pyproject.toml, supporting Turing through Blackwell / RTX 50-series sm_120). CPU also works. Minimum NVIDIA driver: 580.65.06.
  • Model storage for approximately 10 GB under pretrained_models/Fun-CosyVoice3-0.5B/.

Local

# Ubuntu/Debian system packages:
sudo apt-get update && sudo apt-get install -y ffmpeg sox libsox-fmt-all libsndfile1
# Python dependencies (Python 3.10 is managed by uv via .python-version):
uv sync

Then run the service one of two ways:

CommandBehavior
uv run serve run api:cosyvoice_appDeploys against an existing Ray cluster (documented entry point)
uv run python api.pyStarts a local single-node Ray cluster, then serves (the __main__ block)

Both bind the HTTP proxy to SERVE_PORT (default 8000). On first startup, FunAudioLLM/Fun-CosyVoice3-0.5B-2512 downloads from ModelScope into pretrained_models/Fun-CosyVoice3-0.5B/.

Docker

The image uses a multi-stage uv build on nvidia/cuda:13.0.2-cudnn-runtime-ubuntu24.04 (Blackwell / RTX 50-series sm_120 support via PyTorch cu130 wheels). Dependencies are installed by uv in the builder stage and copied into the runtime image. The container entrypoint is start.sh, which starts a single-node Ray head node and then runs python api.py as a non-root user. It sets SERVE_PORT=9998 and CosyVoice3 model defaults.

docker build -t cosyvoice-ray-serve-api .

# CPU
docker run -d -p 9998:9998 --name cosyvoice cosyvoice-ray-serve-api

# GPU (requires the NVIDIA Container Toolkit)
docker run -d --gpus all -p 9998:9998 -p 8265:8265 --name cosyvoice cosyvoice-ray-serve-api

Exposed ports: 9998 (API), 8265 (Ray dashboard), 6379 (Ray GCS), 10001 (Ray client). The container HEALTHCHECK curls http://localhost:9998/v1/model/cosyvoice/healthcheck.

The built-in voice prompts in asset/ are required by /tts and /instruct_tts, so they are copied into the image (the .dockerignore re-includes asset/*.wav). Model weights are not baked in; they download on first start unless you mount a pre-populated pretrained_models/.

These names must match exactly; values are read in the @serve.deployment decorator and reconfigure() in api.py. To change a default permanently, edit api.py.

Health checks and monitoring

  • GET /health and /v1/model/cosyvoice/healthcheck return 200/503 for liveness and readiness probes.
  • Logs are written to stdout (docker logs <container>) and to logs/cosyvoice_api.log.
  • The Ray dashboard is available on port 8265 in the Docker image.

Scaling notes

Each replica loads its own copy of the model and is assigned one GPU when available, so MAX_REPLICAS is effectively bounded by GPU count. Tune TARGET_ONGOING_REQUESTS and MAX_ONGOING_REQUESTS_PER_REPLICA to your latency/throughput needs; long syntheses benefit from streaming responses to reduce time-to-first-byte.
